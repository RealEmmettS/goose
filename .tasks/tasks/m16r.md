## Why

The M16 implementation is in-tree and cross-target checks pass from Windows, but macOS-specific proof cannot be produced on a Windows host. This task owns the host evidence required before claiming full macOS backend readiness beyond implementation closure.

## Acceptance

- `script/smoke_m16_macos.sh` passes on a macOS host.
- `script/package_macos_app.sh` produces `Honk300.app` and validates `plutil`, `codesign`, and `lipo -verify_arch x86_64 arm64`.
- Bundled app launches through `/usr/bin/open -n ... --args start --config <path>`.
- Without Accessibility, overlay/wander/status/reload/stop/audio degrade or work as expected and cursor/window features report denied/unsupported without crashing.
- With Accessibility, cursor nab, foreign-window ride, collect note, collect meme, terminal non-targeting, TUI reload/status, `honk300 stop`, and multi-monitor behavior are manually smoke-tested where hardware allows.
- Evidence covers arm64 native and x86_64 via Rosetta or Intel hardware.
- CI run URL, app artifact names, and Accessibility-granted evidence are recorded in `docs/readiness/m16-m18-readiness.md`.
